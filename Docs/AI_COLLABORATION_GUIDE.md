## 1. AI 협업 원칙

### 1.1 AI를 팀원으로 대하는 방법

AI는 **빠른 초안 제작자 + 반복 작업 자동화 도구** 로 활용한다.
AI의 출력물은 항상 **사람이 검수하고 의사결정** 한다.

```
[기획자/개발자가 방향 결정]
          |
   [AI에게 작업 위임]
          |
  [AI 출력물 검수 & 수정]
          |
   [팀 내 리뷰 & 반영]
```

### 1.2 AI 활용 레벨 정의

| 레벨 | 설명 | 예시 |
|------|------|------|
| L1 - 보조 | 아이디어 초안, 리스트업 | 스킬 이름 아이디어 생성 |
| L2 - 공동 작업 | 구조화 문서 작성 | 스테이지 밸런스 표 생성 |
| L3 - 자동화 | 반복 코드/에셋 생성 | 건물 오브젝트 데이터 일괄 생성 |
| L4 - 검증 | 논리적 오류 감지 | 수식 검증, 밸런스 시뮬레이션 |

### 1.3 역할 분담 기준

| 작업 유형 | 담당 |
|---------|------|
| 핵심 게임성 결정 | 사람 (기획자) |
| 세계관 / 스토리 초안 | AI + 기획자 검수 |
| 밸런스 수치 1차 계산 | AI |
| 밸런스 최종 확정 | 사람 (기획자 + QA) |
| 코드 로직 설계 | 사람 (개발자) |
| 반복 코드 구현 | AI + 개발자 검수 |
| UI 텍스트 / 로컬라이징 | AI + 기획자 검수 |
| 최종 의사결정 | 항상 사람 |

## 2. 프로젝트 컨텍스트 요약 (AI 프롬프트용)

> AI에게 작업을 요청할 때, 아래 컨텍스트 블록을 프롬프트 앞에 붙여 사용하세요.

```
--- [ALIEN CRUSHER 프로젝트 컨텍스트] ---

게임명: ALIEN CRUSHER
장르: 하이퍼 캐주얼 / 디스트럭션 / 탑뷰 3D
플랫폼: 모바일 세로형 (iOS/Android)
세션 길이: 1판 60~90초
핵심 루프: 스테이지 시작 -> 스타터 레인 파괴 -> LANE BREAK -> ROUTE OPEN -> ROUTE HOLD -> ROUTE BONUS / Forward Smash -> 결과 피드백과 성장 선택

[프로덕션 제약 & 디자인 원칙]
- 플레이어 정체성: 외계 문명 Vrak이 만든 파괴 구체(크러셔 볼) 그 자체
- 대부분 에셋(프리팹 포함)은 Unity 기본 기능(Primitive, 기본 Material, 기본 Particle System) 기반으로 제작
- 파괴 연출은 고비용 실시간 메시 파쇄보다 색상 변화 + 파티클 + 카메라/스케일 피드백으로 표현
- 외부 에셋 사용 원칙: FEEL + DOTween + GUI 에셋만 기본 허용 (목표 구현에 불필요하면 미사용)

[주요 시스템 요약]
- 크러셔 볼: 드래그로 조작, 건물 파괴
- 형태/메타 진행: 현재 런타임 명칭은 Sphere, Spike, Ram, Saucer, Crusher이며, 디자인 네이밍 패스 전까지 상태 문서와 프롬프트는 이 명칭을 우선 사용
- 루트 시스템: LANE BREAK, ROUTE OPEN, ROUTE HOLD, ROUTE BONUS, Forward Smash 클러스터 보상, 실패 버킷 기반 결과/로비 추천
- 스킬트리: DP(파괴점수)로 영구 해금, 인게임 레벨업 시 랜덤 3선택
- 재화: DP(무료/파괴 획득), 크레딧(유료), 에너지(스태미나)
- 별 시스템: 목표 점수 50/80/100% = 1/2/3별

[핵심 사용자 감정 목표]
- 파괴의 쾌감 (연쇄 파괴, 슬로우모션 연출)
- 성장의 만족감 (스킬 해금 & 강해지는 느낌)
- 짧고 중독성 있는 반복 플레이

--- [컨텍스트 끝] ---
```

## 3. 작업 분야별 AI 활용 가이드

### 3.1 게임 기획 (Game Design)

#### 언제 AI를 쓰는가
- 스킬 이름/설명 문구 초안
- 스테이지 오브젝트 조합 아이디어
- 아하 모먼트 아이디어 브레인스토밍
- 유사 게임 사례 리서치 요청

#### AI에게 요청하는 방식

**좋은 예시:**
```
컨텍스트 블록 붙여넣기 후...

"W3(산업 단지) 스테이지 10개의 특성을 설계해줘.
각 스테이지는 고유한 기믹 하나씩 가져야 해.
스테이지 번호, 이름, 주요 오브젝트 구성, 핵심 기믹, 목표점수(1/2/3별)를
표 형식으로 정리해줘."
```

**나쁜 예시:**
```
"스테이지 만들어줘" (컨텍스트 없음, 지시 불명확)
```

#### 주의사항
- AI가 제안한 수치는 반드시 스프레드시트에서 **밸런스 검증** 후 사용
- "재미있을 것 같은" 스킬은 반드시 **실제 플레이어 관점에서 재검토**

---

### 3.2 밸런스 설계 (Balance Design)

#### AI 활용 포인트
- 수치 초안 계산 (공식 적용)
- 파라미터 변경 시 영향도 시뮬레이션 (표 형식 요청)
- 극단값(최솟값/최댓값) 시나리오 검증

#### 밸런스 계산 프롬프트 예시

```
컨텍스트 블록 붙여넣기 후...

"아래 공식을 기반으로 W1 스테이지 1~10의 목표 점수(1/2/3별)를
자동 계산해서 표로 만들어줘.

[공식]
- W1 기본 목표 점수 (1별) = 2,000 + (스테이지 번호 x 500)
- 2별 = 1별 x 1.6
- 3별 = 1별 x 2.0
- 퍼펙트 = 3별 x 1.5

결과는 정수로 반올림해줘."
```

#### 밸런스 검수 시 AI 활용

```
"아래 스탯 테이블을 보고 이상한 점을 찾아줘.
특히 연쇄 배율과 착지 배율이 동시에 최대치일 때의 극단값이
3별 조건 대비 너무 쉽거나 어려운 케이스가 있는지 확인해줘.

[스탯 테이블 붙여넣기]"
```

### 3.3 코드 작성 (Unity C# / Game Logic)

#### AI 활용 포인트
- 보일러플레이트 코드 초안
- 반복 데이터 구조 (ScriptableObject) 생성
- 간단한 공식 구현 (점수 계산, 스탯 적용)
- 버그 원인 분석 요청
- FEEL/DOTween 기반 피드백 시퀀스(펀치 스케일, 카메라 셰이크, UI 팝업) 설계

#### 코드 요청 프롬프트 예시

```
"Unity C# 코드를 작성해줘.
크러셔 볼의 드래그 이동 로직인데,
조건은 다음과 같아:

1. 스크린 드래그 입력을 받아서 Rigidbody에 Force로 적용
2. 드래그 거리에 비례해 힘의 크기를 결정 (최대 캡: maxForce 변수로 조절)
3. 공이 맵 경계(BoxCollider)에 닿으면 반발력(bounceCoefficient) 적용
4. Inspector에서 조절 가능하도록 [SerializeField] 사용
5. 이동 중 파티클 이펙트를 재생하는 코루틴 포함

코드에 한국어 주석 추가해줘."
```

#### 스크립터블 오브젝트 일괄 생성 요청

```
"Unity ScriptableObject 데이터 시트를 JSON으로 만들어줘.
건물 오브젝트 데이터 6개 (등급 F~S) 각각에 대해:
- gradeName (string)
- hp (int)
- baseScore (int)
- defenseValue (float)
- debrisCount (int)
- debrisDamageRatio (float, 0~1)

GDD의 수치 기준으로 채워줘."
```

### 3.4 UI/UX 텍스트 & 로컬라이징

#### AI 활용 포인트
- 인게임 팝업 텍스트 작성
- 스킬 설명 문구 통일성 유지
- 다국어 로컬라이징 초안 (한/영/일/중)

#### 텍스트 작성 가이드라인 (AI에게 전달)

```
ALIEN CRUSHER UI 텍스트 작성 원칙:
1. 톤: 강렬하고 파괴적인 SF 느낌, 과장되게 에너지 넘치는 표현 허용
2. 길이: 팝업 제목 10자 이내, 설명 30자 이내
3. 금지어: 지루한, 평범한, 평화로운
4. 이모지: 허용 (아이콘 보완용으로만)
5. 언어 스타일: 짧고 강렬한 동사형 문장 선호

예시 스킬 설명 텍스트:
- 중력 코어: "질량 강화. 부수는 힘이 커진다."
- Ram: "루트를 되찾아라. 밀고 나가라."
```

#### 로컬라이징 프롬프트 예시

```
아래 UI 텍스트를 한국어, 영어, 일본어, 중국어(간체)로 번역해줘.
게임 장르는 SF 파괴 액션이야. 번역도 그 톤을 유지해줘.

[텍스트 리스트]
- PERFECT LANDING
- LANE BREAK -> ROUTE OPEN
- ROUTE HOLD
- ROUTE BONUS -> CLUSTER OPEN
- SMASH CLUSTER OPEN
- ROUTE HOLD MISSED
```

### 3.5 아트 방향 & 에셋 지시서 작성

#### AI 활용 포인트
- Unity 기본 도형 기반 프리팹 조합안 작성 (Cube/Sphere/Cylinder 중심)
- 상태별 색상 팔레트(정상/피격/파괴 직전/파괴 완료) 제안
- 파티클 시스템 프리셋(먼지, 파편, 충격파) 파라미터 초안 작성

#### 아트/파괴 표현 스펙 프롬프트 예시

```
"ALIEN CRUSHER의 W1 오브젝트 6종에 대한 'Unity 기본 기능 기반' 프리팹 스펙 시트를 작성해줘.

제약:
- 외부 3D 모델 에셋 사용 금지 (Unity Primitive 조합 우선)
- 파괴는 메시 파쇄 대신 색상 변화 + 스케일/회전 + 파티클로 표현
- 모바일 성능 우선 (오브젝트당 머티리얼 수, 파티클 최대 수치 포함)

각 오브젝트 항목:
- 프리팹 구성(Primitive 조합)
- 기본 색상 / 피격 색상 / 파괴 완료 색상
- 파티클 타입과 재생 시간
- DOTween/FEEL 연출 포인트 (필요한 경우만)
- 오브젝트 풀링 필요 여부"
```

#### 이미지 생성 AI용 프롬프트 예시 (Midjourney/DALL-E)

```
"ALIEN CRUSHER의 런타임 형태 5종(Sphere, Spike, Ram, Saucer, Crusher)을 위한 아이콘/실루엣 컨셉 참고 이미지 프롬프트를 써줘.
조건:
- 스타일: Low Poly 3D, 밝고 채도 높은 색감
- 구성: 모바일 탑뷰 게임에서 한눈에 구분되는 형태별 실루엣
- 형태: Sphere는 안정적인 기본형, Spike는 관통/약점 공격형, Ram은 돌파/회복형, Saucer는 이동/도달형, Crusher는 중량/압박형
- 배경: 투명 또는 단색 배경, UI 아이콘 제작 참고용
- 색상: 각 형태가 서로 다른 색상 계열을 갖도록 제안
주의: 결과 이미지는 최종 게임 에셋이 아니라 무드/색감 참고용으로만 사용.
Midjourney v6 형식으로 작성해줘."
```

### 3.7 씬 하이러키 & UI 더미 구조 설계

#### AI 활용 포인트
- 제작자 의존성을 낮추는 공통 하이러키 템플릿 설계
- 기능 미구현 상태에서도 플로우가 보이는 UI 더미 화면 목록 도출
- 오브젝트 네이밍 규칙 및 책임 분리 기준 정리

#### 하이러키 더미 생성 프롬프트 예시

```
컨텍스트 블록 붙여넣기 후...

"ALIEN CRUSHER의 Unity 씬 하이러키를 더미 포함으로 설계해줘.

요구사항:
1. 루트는 _App, _Systems, _Gameplay, _UI, _Debug 로 고정
2. _UI에는 HUD, 결과창, 스킬선택, 아웃게임 로비를 더미 패널로 반드시 생성
3. 각 노드마다 역할(1줄), 연결될 스크립트 자리표시자, 초기 활성 상태를 표로 정리
4. FEEL/DOTween을 붙일 권장 지점을 별도 열로 표기

결과는 트리 구조 + 체크리스트 형식으로 작성해줘."
```

## 5. AI 출력물 품질 검수 체크리스트

### 5.1 기획 문서 검수

- [ ] 현재 GDD/로드맵의 LANE BREAK -> ROUTE OPEN -> ROUTE HOLD -> ROUTE BONUS / Forward Smash 루프와 충돌하지 않는가?
- [ ] 수치는 실제 게임 플레이 시간(60~90초) 내에 유효한가?
- [ ] 유저 아하 모먼트에 기여하는가?
- [ ] 유료화 모델과 상충되지 않는가?
- [ ] 기술 구현 가능한 범위인가? (개발자 확인 필요)
- [ ] 에셋/연출 제약(기본 Unity 기능 + FEEL/DOTween/GUI만 허용)을 준수하는가?

### 5.2 밸런스 수치 검수

- [ ] 공식이 GDD 기준과 일치하는가?
- [ ] 최솟값/최댓값 케이스에서 이상한 결과가 없는가?
- [ ] 1별이 너무 어렵거나 3별이 너무 쉽지 않은가?
- [ ] 유료 아이템이 밸런스를 심하게 깨지 않는가?

### 5.3 코드 검수

- [ ] 컴파일 에러 없이 실행되는가?
- [ ] Inspector에서 조정 가능한 변수가 충분히 노출되어 있는가?
- [ ] 성능 이슈 가능성 (Update 루프 내 무거운 연산) 없는가?
- [ ] 모바일 터치 입력을 올바르게 처리하는가?
- [ ] 코드에 한국어 주석이 충분한가?
- [ ] DOTween/FEEL 사용이 과도하지 않고 목적이 명확한가?

### 5.4 텍스트 검수

- [ ] 게임 톤(강렬한 SF 파괴)에 맞는가?
- [ ] 글자수 제한을 지키는가?
- [ ] 오타 및 문법 오류가 없는가?
- [ ] 다국어 번역이 자연스러운가? (원어민 확인 필요)

## 6. 금지 사항 & 주의 사항

### 6.1 AI에게 요청하면 안 되는 것

| 금지 항목 | 이유 |
|---------|------|
| 핵심 게임성 방향 결정 | 기획자의 고유 판단 영역 |
| 유저 데이터 분석 없는 밸런스 최종 확정 | 실측 데이터 필요 |
| 법적/저작권 관련 판단 | 전문 법률 검토 필요 |
| 광고 계약 조건 작성 | 비즈니스 담당자 직접 처리 |
| 보안 관련 코드 (결제, 인증) | 보안 전문가 검토 필수 |
| 허용 범위 외 외부 에셋/플러그인 도입 지시 | 의존성/라이선스/유지보수 리스크 |

### 6.2 AI 출력물 사용 시 주의사항

1. **AI 환각(Hallucination) 주의**: AI가 없는 API, 라이브러리, 기능을 있는 것처럼 작성할 수 있음 → 반드시 실제 존재 여부 확인
2. **수치 계산 오류**: AI는 복잡한 수식에서 계산 실수를 할 수 있음 → 중요 수치는 스프레드시트로 재확인
3. **저작권 주의**: AI가 생성한 코드/텍스트도 외부 소스에서 학습된 것일 수 있음 → 중요 코드는 로직 검토 후 사용
4. **컨텍스트 손실**: 대화가 길어지면 AI가 초기 컨텍스트를 잊을 수 있음 → 중요한 작업 시 컨텍스트 블록을 다시 붙여넣기

### 6.3 AI 도구별 추천 용도

| AI 도구 | 추천 용도 |
|--------|---------|
| Claude (Anthropic) | 기획 문서 작성, 구조화 텍스트, 로직 검토, 장문 분석 |
| ChatGPT (OpenAI) | 코드 초안, 번역, 아이디어 브레인스토밍 |
| Gemini (Google) | 리서치, 다국어 번역, Google Sheets 연동 |
| Midjourney / DALL-E | 아트 레퍼런스 이미지 생성 |
| GitHub Copilot | IDE 내 코드 자동완성 |
